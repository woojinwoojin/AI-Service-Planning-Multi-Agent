# 01. 자료조사 (배경·관련 연구·기술 조사)

> 중간 보고서의 「서론 / 관련 연구 / 기술 조사」 절에 사용.

---

## 1. 문제 배경

### 1.1 문제 정의
서비스 기획서를 작성하려면 **시장·고객·경쟁·환경·리스크**를 폭넓게 조사하고 이를 일관된 문서로 통합해야 한다. 개인이 이를 단독으로 수행하기는 시간·전문성 면에서 어렵다.

가장 흔한 대안인 **단일 LLM에 한 번에 요청**하는 방식은 다음 한계를 가진다.

1. **근거 부재** — 웹 검색을 하지 않으므로 출처 URL이 없다. "그럴듯하지만 확인 불가능한" 초안이 나온다.
2. **환각(hallucination)** — "2022년 조사에 따르면 60%…" 같이 **근거 없는 통계를 지어냄**.
3. **품질 통제 불가** — 생성 과정이 하나의 블랙박스라 어느 단계가 부실한지 짚어 개선할 수 없다.

> 핵심 통증은 "그럴듯함"이 아니라 **"믿을 수 있는가(추적 가능한가)"** 이다.

### 1.2 접근 방향
단일 호출을 **역할이 분리된 여러 Agent의 파이프라인**으로 나눈다. 각 단계가 **중간 산출물과 근거(출처)를 남기며**, 조사 → 분석 → 작성 → 검토 → 검증을 통제 가능한 형태로 수행한다. 이로써 (1) 실제 웹 검색으로 근거를 확보하고, (2) 단계별로 품질을 점검·재작성하며, (3) 검증의 범위와 한계를 사용자에게 정직하게 표기한다.

## 2. 기존 방식 조사 및 비교 (실측)

동일 주제·동일 서식·동일 심판(LLM)으로 **① 검색 없는 단일 프롬프트 1회**와 **② 근거 기반 Multi-Agent 파이프라인**을 비교하였다. (원자료: `../docs/comparison_result.md`, `../docs/multimodel_result.md`)

### 2.1 메인 비교 (6주제 · 심판 3회 평균 · gpt-4o-mini)

| 지표 | 단일 프롬프트 | Multi-Agent |
|---|---|---|
| 기획서에 포함된 **고유 출처 URL 수**(객관) | **0** | **5** |
| LLM 심판 총점 | 84.9 | **86.5 (+1.6)** |
| 주제별 우위 | — | **5 / 6 주제** (1개 −0.3 근소 열세) |

### 2.2 생성 모델을 바꿔도 유지되는 차이 (3주제 · 방향성 확인)

| 생성 모델 | 단일 | Multi | 점수차 | 출처(단일/Multi) |
|---|---|---|---|---|
| gpt-4o-mini | 84.8 | 86.6 | +1.8 | 0 / 5 |
| gpt-4o | 84.7 | 86.3 | +1.6 | 0 / 5 |

**해석**: 모델이 강해지면 점수 격차는 줄어들 수 있으나(단일도 글을 잘 씀), **출처 격차 0 vs 5는 그대로다**. "강한 모델도 스스로 검색하지는 않는다"는 **구조적** 차이. 
※ 정직성: 출처 수는 "포함 개수"이며 URL 사실성 검증은 아니다. 3주제 표는 **일반화가 아닌 방향성 확인**.

## 3. 사용 기술 조사

| 영역 | 채택 기술 | 선택 이유 |
|---|---|---|
| Agent 오케스트레이션 | **LangGraph** (`StateGraph`) | 상태(State) 기반 그래프로 **직렬/병렬 구조를 명시적으로** 정의·비교 가능. 노드별 fan-in/fan-out, reducer로 병렬 상태 병합 지원 |
| LLM | **OpenAI(gpt-4o-mini)** 실사용 · 타 provider(Anthropic 등) **지원 가능** | provider 교체 가능하게 추상화. 데모는 비용·속도 균형상 gpt-4o-mini(Anthropic은 미사용) |
| 웹 검색(grounding) | **Tavily** | LLM 친화적 검색 API. 실제 출처 URL·요약 스니펫 확보 → 근거 추적성의 핵심 |
| API 서버 | **FastAPI** | 비동기·자동 문서화(OpenAPI). **SSE 스트리밍**으로 진행 상황 실시간 전송 |
| 관측성 | **Langfuse** | 노드별 latency·토큰·비용 트레이스 → "느낌이 아닌 측정"으로 병목 진단 |
| 저장 | **SQLite** | 별도 서버 없이 실행 이력(프로젝트·상태 JSON) 로컬 저장 |
| 문서 출력 | **python-docx / python-pptx** | 기획서를 Word·PowerPoint로 내보내기 |
| 품질 | **pytest(138) · ruff · GitHub Actions CI** | LLM 미호출 결정론적 회귀 테스트 + PR마다 자동 검사 |

### 3.1 관련 개념
- **RAG / Grounding**: 검색으로 얻은 외부 근거를 프롬프트에 주입해 사실성·추적성을 높이는 기법. 본 프로젝트는 검색 스니펫을 `<검색결과>` 구획으로 격리해 **프롬프트 인젝션**을 방어한다.
- **Multi-Agent 워크플로**: 자율 협상형이 아니라, 동일 LLM을 **역할별(목표·도구·입출력 스키마가 다른) 소프트웨어 구성요소**로 분리한 **역할 특화 파이프라인**.
- **정보 신뢰성 3-Tier**: (Tier 1) 검증 범위·한계 투명 표기 → (Tier 2) 주장 유형 분류·주장별 근거 연결 → (Tier 3) URL 원문 대조. 현재 Tier 1 완료, Tier 2 진입 준비(Evidence Registry).
