# 02. 요구사항 명세서 (SRS)

> 중간 보고서의 「목표 및 범위 / 요구사항 분석」 절에 사용.
> 표기: `구현 상태` = ✅ 완료 · 🚧 진행 · 🔜 예정. 실제 코드 기준(2026-07-23).

---

## 1. 목표 및 범위

### 1.1 목표
한 줄 아이디어 입력을 받아, 여러 역할 특화 AI Agent가 **웹 근거를 수집하고 단계별로 검토**하여 **출처가 인용된 14개 섹션 서비스 기획서**를 생성한다.

### 1.2 범위
- **포함**: 근거 수집(웹검색) → 다각도 분석 → 기획서 작성 → 평가·자동 재작성 → 일관성 편집 → 근거 검증 → 다운로드(MD/JSON/DOCX/PPTX) → 이력 저장 → 사용자 수정.
- **제외(현 단계)**: URL 원문 사실성 재검증(Tier 3), 인증/과금, 프로덕션 배포.

### 1.3 사용자 스토리
- US-1: 기획 초보자로서, 아이디어 한 줄만 넣어도 나머지 항목을 **추천**받아 빠르게 시작하고 싶다.
- US-2: 사용자로서, 생성된 기획서의 **근거(출처)** 를 확인하고 신뢰도를 판단하고 싶다.
- US-3: 사용자로서, 마음에 안 드는 부분을 **직접 수정 요청**해 다시 받고 싶다.
- US-4: 사용자로서, 결과를 **Word/PPT로 내려받아** 실제 발표·공유에 쓰고 싶다.
- US-5: 운영자로서, 실행 **비용·시간·실패**를 관측해 품질을 관리하고 싶다.

## 2. 기능 요구사항 (Functional Requirements)

| ID | 요구사항 | 관련 컴포넌트 | 상태 |
|---|---|---|---|
| FR-01 | 아이디어(프로젝트명·설명·타깃·문제·키워드) 입력을 받는다 | `ProjectInput`, UI 입력 화면 | ✅ |
| FR-02 | 프로젝트명 기반으로 **빈 항목만** AI가 자동완성하고 사용자 입력은 보존한다 | `/suggest`, `suggest.py` | ✅ |
| FR-03 | 자동완성은 **내 값 vs AI 제안 비교** 모드와 **추천 이유·확신도** 를 제공한다 | `/suggest?compare`, UI | ✅ |
| FR-04 | **웹 검색(Tavily)** 으로 시장·경쟁 근거를 수집하고 출처를 확보한다 | `research.py`, `competitor.py`, `search.py` | ✅ |
| FR-05 | 시장·고객·경쟁·PESTEL·SWOT·수익모델·리스크를 분석한다 | `agents/*.py` | ✅ |
| FR-06 | **고정 14개 섹션** 기획서를 작성하고 실제 출처만 「참고자료」로 인용한다 | `draft_writer.py` | ✅ |
| FR-07 | Reviewer가 100점 평가하고, 총점 90 미만이면 **1회 자동 재작성** 한다 | `reviewer.py`, `workflow.py` | ✅ |
| FR-08 | 일관성 편집(Polish) 후 **최종본을 재평가**해 표시 점수를 일치시킨다 | `draft_writer.polish`, `reviewer.final_reviewer` | ✅ |
| FR-09 | 기획서 주장을 수집 근거와 대조해 **근거 확인율**을 산출한다 | `verifier.py` | ✅ |
| FR-10 | 실제 검색 근거를 **단일 Evidence Registry**로 통합한다 | `evidence.py` | 🚧 (2-1 착수) |
| FR-11 | 진행 상황을 **SSE로 실시간** 전송하고 진행률·ETA를 표시한다 | `/run/stream`, UI | ✅ |
| FR-12 | 사용자의 **수정 요청을 반영해 재작성**하고 재평가한다(Human-in-the-Loop) | `/revise` | ✅ |
| FR-13 | 실행 결과를 **이력(SQLite)** 에 저장하고 목록·상세 조회한다 | `store.py`, `/projects` | ✅ |
| FR-14 | 결과를 **Markdown/JSON/DOCX/PPTX** 로 내보낸다 | `/export/docx`, `/export/pptx`, `/run/save` | ✅ |
| FR-15 | 실행 **구조(직렬/병렬)** 를 선택·비교할 수 있다 | `WORKFLOW_MODE`, `workflow.py` | ✅ |
| FR-16 | 실패/fallback 노드를 판정해 **실행 품질(run_status)** 을 표면화한다 | `_assess_quality` | ✅ |

## 3. 비기능 요구사항 (Non-Functional Requirements)

| ID | 분류 | 요구사항 | 근거/구현 | 상태 |
|---|---|---|---|---|
| NFR-01 | 신뢰성 | 개별 Agent가 실패해도 파이프라인은 **처음~끝 완주** 한다 | `_safe()` 래퍼 + 노드별 fallback(`_dummy`) | ✅ |
| NFR-02 | 정직성 | 검증은 "수집 근거와의 일치"이며 **URL 원문 사실검증이 아님**을 UI·문서·JSON에 동일 표기 | `reliability.py`, `verification_scope` | ✅ |
| NFR-03 | 성능 | 독립 분석 단계 병렬화로 실행시간을 단축한다 | 병렬 −16.1%(127.4→106.8s), 분석 −41% | ✅ |
| NFR-04 | 관측성 | 실행마다 토큰·비용·지연·fallback을 집계·표시한다 | `usage.py`, `timing.py`, Langfuse | ✅ |
| NFR-05 | 재현성 | 실험은 git commit·모델·온도·프롬프트·rubric 버전으로 **서명** 한다 | `experiment_signature`, baseline-v1 | ✅ |
| NFR-06 | 보안 | 검색 결과를 `<검색결과>` 로 격리해 **프롬프트 인젝션**을 방어한다 | `research.py`, `competitor.py` | ✅ |
| NFR-07 | 이식성 | LLM provider를 교체 가능하게 추상화한다(OpenAI 실사용 · Anthropic 등 지원 가능 · 더미 모드) | `llm.py` | ✅ |
| NFR-08 | 품질 | LLM 미호출 결정론적 회귀 테스트 + PR마다 CI 자동 검사 | pytest 138 · ruff · GitHub Actions | ✅ |
| NFR-09 | 사용성 | 2분 내외 대기 동안 진행률·단계 언어로 상태를 안내한다 | SSE 진행 화면 | ✅ |

## 4. 제약 조건 및 가정
- LLM/검색 API 키가 없으면 **더미 모드**로 흐름만 검증(무비용).
- 크레딧 제약상 실측 실험은 소표본(6주제/3주제)으로 수행.
- 저장소가 OneDrive 내부라 git 파일 잠금이 간헐 발생(재시도로 대응).
