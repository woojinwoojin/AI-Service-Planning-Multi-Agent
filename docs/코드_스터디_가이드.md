# 코드 스터디 가이드

> 이 코드베이스는 **"한 가지 패턴이 12번 반복"**되는 구조라, 뼈대 → 패턴 하나 → 엔진 순으로 보면 전체가 한 번에 뚫린다.

## 핵심 관점 하나

```
┌─ 바깥(신뢰 불가) ─┐        ┌─ 안(신뢰) ─┐        ┌─ 바깥 ─┐
  HTTP 요청 JSON    →  Pydantic  →  ProjectState  →  Pydantic  →  HTTP 응답
  (ProjectInput)      검증·파싱     (그냥 dict)      직렬화       (RunResult)
```

**경계에서는 검증(Pydantic), 안에서는 신뢰(TypedDict).** 내부 State는 런타임 검증이 없으니 **각 Agent가 `_validate()`로 스스로 계약을 지킨다.**

## 에이전트 표준 패턴 (한 번 익히면 12개가 같음)

모든 `app/agents/*.py`는 5부 구조:

1. **스키마 상수** — 출력 키·타입
2. **`_validate(result, fallback)`** — LLM 출력을 스키마에 맞게 강제(다음 Agent를 위한 계약)
3. **`_dummy(si)`** — 입력을 반영한 "구조가 온전한" fallback
4. **노드 함수** — State 읽기 → 프롬프트 구성 → `llm.complete_json(status=…)` → `_validate` → `mode_label`로 로그
5. **부분 dict 반환** — `{"xxx_result": ..., "logs": ...}` → LangGraph가 State에 merge

---

## 정직성 파이프 (한 줄기로 이어짐)

```
llm.py: 실패 → _flag(status, 원인)         (혼잡/연결/형식/처리)
   → agent: mode_label(status) → "[node] … (fallback·혼잡)" 로그
   → workflow._assess_quality: 로그 정규식 파싱 → run_status / fallback_reasons
   → routes: RunResult에 실어 응답
   → 프론트: 섹션별 원인 안내·환각 경고 표시
```

## ⚠️ 특히 주의 깊게 볼 부분

1. **contextvar와 스레드 전파** — `usage.py`/`demo.py`. LangGraph가 노드를 스레드로 돌려서, 스트리밍(anyio 스레드풀)에선 contextvar 전파가 안 되는 함정이 있다(그래서 `demo.py`는 노드 진입 시 state에서 직접 읽는다).
2. **fallback이 "구조는 유지"하는 이유** — `_validate`가 누락 키를 빈 값으로 채워 다음 Agent가 항상 온전한 입력을 받는다 → 중간이 실패해도 완주.
3. **점수 3형제** — `review_result`(초안, 재작성 판단용) / `initial_review_result`(기록) / `final_review_result`(최종본 재채점, 화면 표시).
4. **재시도 2겹** — `_invoke_with_retry`(네트워크) vs `complete_json` 루프(JSON 파싱 재프롬프트).
5. **검증기의 한계** — `verifier.py`는 탐지·보고만 하고 문서를 고치지 않는다(정직하게 명시). 환각 대응 Layer 2가 그 결과를 프론트에서 경고로 표시.
6. **프롬프트 = 코드** — `app/prompts/templates.py`의 `NO_FABRICATION_RULE`·`UNTRUSTED_SEARCH_GUARD`·"앞 단계 결과만 근거로"가 동작의 절반을 결정.
