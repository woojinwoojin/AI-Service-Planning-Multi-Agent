| 단계 | 직렬(ms) | 병렬(ms) |

- 병렬화 효과는 `analysis_block`에 집중(35.7s→21.0s, **-41%**). 분석 4분기를 병렬 실행한 결과.
  → 정확한 표현: **병렬화 대상인 분석 구간에서 41% 지연 감소를 달성했고, 전체 파이프라인 기준으로는 16.1% 개선**. 전체 개선폭이 목표(20%)에 못 미치는 것은 병렬 구현 부족이 아니라, 파이프라인의 절반 이상이 순차 문서 생성 구간이기 때문(구조적 상한).
- **다음 병목은 문서 재생성 구간**: `revise_or_finalize`(24.4s) + `polish`(17.0s) + `draft`(16.7s) = 병렬 기준 **약 58s로 전체 106.8s의 절반 이상**. 병렬화로 줄지 않는다(순차·단일 문서를 3회 생성/재생성). → Phase 2 PR-7(섹션 단위 수정)의 타깃.
- coverage=1.0 → 측정 단계가 전체 wall time을 사실상 100% 설명(프레임워크 오버헤드 무시 가능).

### 재작성(revision) 실행률

자동 재작성은 조건부다: 초기 Reviewer 총점이 `PASS_SCORE` 미만이고 `revision_count < 1`일 때만 1회 실행, 아니면 재작성 없이 finalize(문서 복사, LLM 호출 없음).

| 지표 | 직렬 | 병렬 |
|---|---|---|
| 재작성 실행 | 12/12 | 12/12 |
| **재작성 실행률** | **100%** | **100%** |
| revise_or_finalize 중앙값(ms) | 24,471 | 24,435 |
| revise_or_finalize 최소(ms) | 21,751 | 19,612 |

> 확정 근거: 24회 전부 LLM 호출 수가 13으로 **균일**하고 `revise_or_finalize` 단계가 모두 **≥19.6s**(finalize였다면 ~0ms) → 전 실행이 revise 분기를 탔다(초기 점수가 매번 PASS_SCORE 미만).
>
> **PR-7 함의**: 재작성이 100% 실행되고 매번 문서 전체(~24s)를 재생성하므로, **섹션 단위 수정의 절감 잠재력이 매우 크다**. (실행률이 낮았다면 효과가 제한적이었을 것.)

24회 전체 주장 분포: **supported 185 · unsupported 32 · uncertain 0** (총 217건) → 근거 확인율 **85.3%**.

**모드별 분해** (집계만 보면 모드 간 차이가 숨는다):

| 지표 | 직렬 | 병렬 |
|---|---|---|
| supported | 93 | 92 |
| unsupported | 13 | 19 |
| **근거 확인율** | **87.7%** | **82.9%** |

- 모드 간 ~5pp 차이는 **LLM 비결정성**(temperature=0.3)으로 본다. 병렬화는 Agent 입력·프롬프트를 바꾸지 않으므로(실행 순서만 다름) 근거율에 구조적 영향을 주지 않는다. 표본(모드당 n=12)이 작아 이 차이를 병렬화 효과로 일반화하지 않는다.

- 범위 주의: 검증은 `search_snippet_only`(실제 검색 스니펫 대조)이며, 원문 대조(Tier 3)는 미포함(`docs/정보신뢰성_전략.md`).

## 기준선 해석 (개편 판단 기준)

- 병렬화는 **품질을 떨어뜨리지 않고**(구조 100%·fallback 0·근거율 유지) wall time을 -16.1% 줄였다 → 비열등성 + 성능 개선 확인.
- 이후 개편의 성공 판단: 이 표 대비 **구조 품질·근거율 하락 없이** 병목(문서 재생성) latency를 줄이거나, 근거율·내용 품질을 올렸는가.
- **이 baseline은 '구조·성능' 기준선이다.** "논리적으로 좋은 기획서인가·차별성이 구체적인가·수익모델이 현실적인가·실제 업무에 쓸 수 있는가"는 증명하지 못한다. 두 축을 나란히 봐야 한다:
  - **구조·성능 벤치**(이 문서) → *빠르고 오류 없이 완성됐는가*
  - **내용 품질 평가**(`docs/평가_루브릭_v1.md`, 8기준 LLM 심판) → *실제로 좋은 기획서인가*

## Phase 2 (PR-7 섹션 단위 수정) 성공 기준 — 대 baseline-v1

> 아래는 **목표치**(측정 결과 아님). 개편 후 동일 벤치로 이 표와 비교한다.

**성능**
- `revise_or_finalize` 중앙값 ≥ 40% 감소 (24.4s → ≤ ~14.7s)
- 전체 wall time ≥ 10% 추가 감소 (병렬 106.8s → ≤ ~96s; 개편 前 직렬 대비 ~24%)
- 평균 토큰 ≥ 10% 감소 · 호출 비용 증가 없음

**품질(비열등성)**
- 14섹션 완성률·순서·PESTEL 표 정상률 **100% 유지**
- fallback 0 유지 · 근거 확인율 **≥ 85.3% 유지**(모드별 기준 포함)
- **수정되지 않은 섹션은 원문 보존** · Reviewer 지적 이슈 **해결률** 측정

**안정성**
- 섹션 수정 실패 시 **full-revise fallback**
- 재채점이 이전보다 나빠지면 **기존 최고 버전 유지**
- 수정 섹션 수·수정 전략을 State에 저장(관측성)
