## Phase 1 — 평가 기준선 구축 (+ 기본 CI 즉시)  ★개편 前 평가

**목표:** 구조를 크게 바꾸기 *전에*, "정말 좋아졌는가"를 측정할 최소 평가 기반을 만든다.

**작업:**
- **고정 평가 세트 10~20개 주제**(현 `run_compare` 6개 확장) + 시드 고정.
- **루브릭 명문화**: 문제 정의 · 고객 구체성 · 시장 분석 · 경쟁 차별성 · 수익 모델 · 실행 가능성 · 논리 일관성 · 근거 활용.
- **결정론적 구조 검사 재사용**: `parallel_bench.structural_quality`(14섹션·순서·빈 섹션·PESTEL 표·출처 수·검증 분포)를 평가 파이프라인에 통합.
- **비용·latency·버전 기록** + 결과 버전 저장(실험 서명).
- **사람 기준선(소규모)**: 사람이 쓴 좋은 기획서 3~5건과 모델 결과를 비교해 **점수 기준 보정**. → `90점` 절대 신뢰 금지.
- **평가 모델 변동성 측정**: 같은 문서 N회 채점의 분산 기록.

**[정정]** 이미 있는 자산: `run_compare.py`(LLM judge), `structural_quality`, `experiment_signature`. Phase 1은 신규 구축이 아니라 **이들을 '평가 세트+루브릭+사람 보정'으로 승격**하는 작업.

**완료 게이트:** 고정 세트·루브릭·사람 보정 기준이 커밋되고, `eval` 리포트가 기준선과 함께 재현 가능.

**CI (지금 바로 병행):** 아래 '상시 트랙 A' 참조. Phase 1과 동시에 GitHub Actions 도입.

## Phase 2 — Multi-Agent 구조 개혁 (가장 큰 백엔드 개편)

> 동적 Supervisor를 한 번에 만들지 않는다. '자유 대화'가 아니라 **작업(Artifact)과 근거(Evidence)를 주고받는 통제 가능한 협업**으로 간다.

- **2-1 Evidence Registry** — 모든 실제 검색 결과를 공통 저장소로: `{evidence_id, source_agent, query, url, snippet, used_by_claims[]}`.
  - [현황] `research.source_objects` + `competitor_sources`가 이미 부분 근거. 이를 **단일 레지스트리로 통합**하는 작업.
- **2-2 Artifact Contract** — 각 Agent 결과를 표준 객체로: `{artifact_id, owner_agent, artifact_type, evidence_ids[], content}`.
- **2-3 Reviewer Issue 구조화** — 점수+자유 피드백 대신 `{target_section_id, issue_type, severity, recommended_agent, revision_instruction}`.
- **2-4 섹션 단위 수정** — 전체 재작성 대신 문제 섹션만 담당 Agent가 보완. **full-revise fallback 유지**.
- **2-5 제한된 동적 실행** — Agent가 근거 부족을 '발견했을 때만' Research 추가 호출(예산 상한 안에서).

**[통합]** 2-3 + 2-4는 이미 상세 계획이 있는 **PR-7**(`docs/병렬화_측정결과_및_PR7_계획.md`)과 동일 축이다. 순서 제안: **2-1 Evidence Registry → 2-2 Artifact → (PR-7) 2-3/2-4 → 2-5**. Evidence Registry를 먼저 해야 2-3의 `evidence_id` 연결과 Phase 3(신뢰도)가 재작업 없이 이어진다.

**완료 게이트(단계별):** 각 하위 단계 후 **Phase 1 평가 세트로 비열등성 확인**(구조 품질·비용 하락 없음) + baseline 대비 latency 기록.

## Phase 3 — 정보 신뢰도 고도화 (Evidence Registry와 **함께** 설계)

> 신뢰도는 개편 후 붙이는 부가기능이 아니라 Evidence Registry와 같은 축. Tier 1은 유지.

- **Tier 2**: 문장을 `fact | inference | proposal`로 분류 → **사실 주장만** 검증 → 각 주장에 `evidence_id` 연결 → '근거 미확인'과 '반대 근거'를 분리 → 소규모 Ground Truth 평가.
  - [현황] `verifier`에 `verification_scope`·claim `basis`·supported/unsupported/uncertain 이미 존재 → Tier 2 기반 일부 확보.
- **Tier 3**: URL 원문 Extract → 주장 ↔ 원문 문장 직접 비교 → 최신성 검사 → 독립 출처 여부 → 출처 간 충돌 탐지.

**추천 순서(중요):** Evidence Registry(2-1) → 주장-Evidence 연결(2-3/Tier2) → 원문 Extract → 정식 사실성 검증(Tier3). **Tier 3를 Evidence 구조 전에 하면 나중에 다시 갈아엎게 된다.**

**완료 게이트:** Tier 2 = 주장별 `evidence_id` 연결률·사실 주장 검증률이 리포트에 표기. Tier 3 = 원문 대조 표본에서 정확도 측정.

## Phase 4 — 평가 프로세스 최종 (서비스용 품질 게이트)  ★개편 後 평가

**목표:** 단순 점수가 아니라 **출력 가능 여부를 결정하는 게이트**.

```python
release_ready = (
    score >= calibrated_threshold
    and critical_issues == 0
    and major_issues <= 1
    and structure_valid
    and evidence_coverage >= 0.8
)
```

**작업:** 수정 전후 후보 버전 비교 → **나빠지면 이전 최고 버전 유지**(현재 final_reviewer 재채점은 있으나 '최고 버전 채택'은 신규) · 최대 수정 라운드 제한 · 평가 모델 변동성 측정 · **Writer/Reviewer 역할(모델) 분리** · 사람↔LLM 평가 상관 측정 · **미해결 이슈를 사용자에게 표시**.

**완료 게이트:** 게이트 통과/미통과가 State·응답에 표면화되고, unresolved issue가 UI 계약에 포함.

## Phase 7 — E2E · 보안 · 부하 · 사용자 테스트

- **E2E**: 입력→생성→수정→다운로드 전 흐름 자동화.

### B. 관측성·운영 모니터링
usage/latency 확장: Agent별 성공·실패율 · 모델별 토큰/비용 · 검색 실패율 · 품질 점수 분포 · 평균 수정 횟수 · 90점 미만 비율 · 요청별 Trace. (Langfuse는 이미 배선되어 있어 여기서 효과 큼.)

### C. 모델·프롬프트 버전 관리 / 재현성
결과 원인 추적을 위해 실험 서명에 포함: `git commit · workflow version · prompt version · model · temperature · 검색 설정 · rubric version`.
> [현황] `experiment_signature`에 git/model/topics/version 일부 존재 → **prompt/rubric version 추가**로 확장. 프롬프트 변경 후 옛 벤치와 직접 비교 금지.

### D. 비용·시간 예산 정책
실행별 상한: 최대 병렬 Agent 수 · 최대 추가 검색 · 최대 수정 라운드 · 최대 LLM 호출 · 최대 비용 · 최대 wall time. (동적 실행(2-5) 도입 전 **반드시** 선행.)

### F. 문서 동기화
`docs/ARCHITECTURE.md`(뒤처짐)·README를 각 구조 개편 후 갱신. 발표/리뷰 전 필수.
