# 정보 신뢰성 및 검증 투명성 전략 (구현 계획)

> 출처 문서: `information_reliability_summary.md` (신뢰성 설계·평가 이론 정리)
> 이 문서의 목적: 그 이론을 **현재 코드베이스에 매핑**하고, 발표 일정 안에서
> "무엇을 / 왜 / 어떤 순서로 / 어디까지" 할지 결정한다. **아직 구현하지 않는다.**
> 개정(2026-07-21): 리뷰 피드백 3건 반영 — ①검증 상태와 주장 유형 분리,
> ②"신뢰도 등급" → "출처 유형/권위성 힌트", ③Tier 1의 정체 = 신뢰성 향상이 아닌 **검증 투명성** 향상.

## 0. 한 줄 결론

> 지금 시스템은 **근거 추적성(URL 존재)** 은 있지만 **사실성 검증**은 없다.
> 발표까지는 새 팩트체크 기능을 급하게 붙이지 않고, **현재 시스템이 수행한 검증의
> 범위와 한계를 사용자에게 정확히 전달**하는 데 집중한다.
> 실제 원문 재검증(Extract·GT)은 **발표 후 로드맵**으로 명시한다.

이는 메모리에 기록된 발표 프레이밍(**출처 0 vs 5 헤드라인**, 옵션 A 확정)과 일치한다.
핵심 메시지를 "URL 수가 많다"에서 "**우리는 우리 검증의 한계를 안다**"로 한 단계 끌어올린다.
Tier 1이 실제 사실 정확도를 크게 올리지는 못한다 — **잘못된 확신(오해)을 줄이는 것**이 목표다.

## 1. 현재 코드가 실제로 하는 일 (사실 확인)

| 영역 | 파일 | 실제 동작 | 판정 |
|---|---|---|---|
| 웹 검색 | `app/services/search.py` | Tavily `search_depth="basic"`, 결과 = `{title, url, content(snippet)}` | **근거 후보 수집기** (사실 판정기 아님) |
| 근거 주입 | `app/agents/research.py` | snippet을 `<검색결과>`로 감싸 프롬프트 주입, 실제 URL을 `sources`에 병합 | 근거 추적성 O, 원문 재접속 X |
| 검증 | `app/agents/verifier.py` | 기획서 주장 ↔ `research_result`(텍스트) 대조, `supported/unsupported/uncertain` | **내부 근거 일치성 검증** (외부 사실성 검증 아님 — docstring에 이미 정직하게 명시됨) |

**중요:** `verifier.py`는 URL 원문에 재접속하지 않는다. `search.py`는 snippet만 쓰고
원문(full text)을 Extract 하지 않는다. 즉 요약 문서 §3의 "확인할 수 없는 것"이
그대로 현재의 공백이다. 이 전략은 그 공백을 **숨기지 않고 표시**하는 것에서 출발한다.

### 1-1. 숨은 구현 조건 (배지 표시 전 반드시 해결)

`research.py`의 `_merge_sources()`는 출처를 **`"제목 — URL"` 문자열**로 합쳐 저장한다.
따라서 **출처 유형 배지를 표시하려면 URL 문자열만으로는 부족**하고, 각 출처를
객체로 유지해야 한다. 최소 스키마:

```json
{ "title": "...", "url": "...", "snippet": "...", "source_type": "government" }
```

변경 경로: `search.py`(분류값 부여) → `research.py`(문자열 병합 대신 객체 유지) →
`State`/결과 스키마까지 구조 전달. **이것이 Tier 1에서 가장 큰 코드 변경이다.**

## 2. 요약 문서 개념 → 코드 매핑

요약 문서 §8(검증 아키텍처)의 각 단계를 현재 코드 기준으로 분류:

| 요약 §8 단계 | 현재 상태 | 필요한 파일 | 난이도/비용 |
|---|---|---|---|
| 8.1 주장 유형 분류 (사실/추론/제안) | 없음 | verifier 스키마 확장 | 중 (기존 콜 재사용) |
| 8.2 출처 후보 수집 | **있음** (`search.py`) | — | — |
| 8.3 출처 유형 분류 (도메인 기반) | 없음 | `search.py`에 규칙 함수 | **하 (규칙 기반, LLM 불필요)** |
| 8.4 URL 원문 Extract | 없음 | `search.py`에 `extract()` | 중 (Tavily Extract API, 비용↑) |
| 8.5 주장-근거 대조 (support/contradict…) | 부분 (텍스트만) | `verifier.py` 라벨 확장 | 중 |
| 8.6 출처 독립성 판정 | 없음 | 신규 | 상 (발표 후) |
| 8.7 최종 검증 상태 표시 | 부분 (3단계) | verifier + export | 하 |
| §9 GT Benchmark | 없음 | `data/gt/` + eval 스크립트 | 상 (발표 후) |

### 2-1. 개념 정리 — 세 축을 섞지 않는다

리뷰 지적의 핵심: `분석적 추론`·`서비스 제안`(문장 **유형**)과
`단일 출처`·`확인 불가`(근거 **상태**)는 서로 다른 기준이다. 하나의 status로 합치면
"AI 시장은 성장할 것으로 예상된다"처럼 **추론이면서 단일 출처인** 문장이 꼬인다.
따라서 세 축으로 분리한다(전량 구현은 Tier 2에서):

```json
{ "claim_type": "fact | inference | proposal",
  "evidence_status": "supported | contradicted | insufficient",
  "source_count": 1 }
```

## 3. 3-Tier 전략 (발표 기준선으로 절단)

### Tier 1 — 투명성 기준선 (발표 전 필수, LLM 추가 호출 0)
**목표: 사실성을 새로 보장하는 단계가 아니라, 현재 시스템이 수행한 검증의
범위와 한계를 사용자에게 정확히 전달하는 단계.**

1. **검증 라벨 표현 정교화** (`verifier.py`)
   - 기존 `supported/unsupported/uncertain`을 **더 정직한 표현**으로 표면화:
     - `supported` → **수집된 검색 근거에서 확인됨**
     - `unsupported` → **수집된 검색 근거에서 확인되지 않음** (⚠️ "거짓/틀림"으로 표현 금지 —
       원문을 안 봤으므로 snippet에서 못 찾았을 뿐일 수 있음)
     - `uncertain` → **현재 근거로 판단 불가**
   - LLM에 "확정하지 말고 모르면 판단 불가로 두라"를 강제 (허위 통과율↓).
   - ⚠️ **주장 유형(사실/추론/제안) 분류는 여기 넣지 않는다 → Tier 2.**
2. **검증 범위(verification_scope) 명시**
   - 각 검증 결과에 `"verification_scope": "search_snippet_only"` 부착.
   - 사용자 문구: *"이 판정은 수집된 검색 요약문과의 일치 여부이며, URL 원문의
     사실 확인을 의미하지 않습니다."* (일반 면책 문구보다 구체적)
3. **출처 유형 분류 (규칙 기반)** (`search.py`)
   - `classify_source_type(url)` — 도메인으로 유형 태깅. LLM 호출 없음(비용 0).
   - ⚠️ **"신뢰도 등급"이 아니라 "출처 유형/권위성 힌트"** — 정부 자료도 오래될 수 있고,
     기업 자료는 편향, 언론이 항상 정확한 것도 아니므로 절대적 신뢰 판정으로 부르지 않는다.
   ```python
   SOURCE_TYPES = {
       "government": "정부·공공기관",
       "academic":   "학술·연구기관",
       "corporate":  "기업 공식자료",
       "news":       "언론",
       "community":  "블로그·커뮤니티",
       "unknown":    "기타",
   }
   ```
   - 화면 표기: `공식·공공 출처 2건 · 학술 1건 · 언론 2건` (❌ "고신뢰 출처 3건" 금지 —
     사실 검증 완료로 오해됨). **선행 조건: §1-1 출처 객체화.**
4. **동일한 한계 문구를 3곳에 일관 표시** (`markdown_export.py` 만으로는 화면 사용자가 못 봄)
   - UI 결과 화면 / Markdown·DOCX 내보내기 / 실행 결과 JSON.
   - 결과 스키마에 단일 메타데이터를 두어 재사용:
   ```json
   { "verification_summary": {
       "scope": "search_snippet_only",
       "original_document_checked": false,
       "fact_check_completed": false } }
   ```
5. **발표 자료에 검증 가능/불가능 범위 명시** — 요약 §14의 Q1~Q6 답변을 `발표자료_1차.md`에 삽입
   (개발 항목이라기보다 **문서화 작업**; 다음 발표가 있을 때 유지).

> Tier 1은 **코드 리스크 낮음 + 발표 방어력 최고**. 여기까지가 발표 필수선.
> "그럼 신뢰도가 실제로 올라간 건가요?"에 방어 가능해진다.

### Tier 2 — 탐색적 검증 (시간 남으면, 발표 전 선택)
6. **주장 유형 분류** (§8.1) — `claim_type`(fact/inference/proposal)과
   `evidence_status`를 **분리**해 verifier 기존 출력 스키마에 필드 추가(별도 LLM 콜 대신 재사용).
   프롬프트·`_validate`·UI가 함께 바뀌므로 Tier 1보다 구현 위험 조금 높음.
7. **출처 수·유형을 claim 단위로 연결**.
8. **소규모 GT 스모크셋 (10건)** — §9의 30~50건 대신 발표 데모 주제 한정.
   - **표본 수를 함께 보고**: `허위 통과 1/5 · 판단 불가 2/5 · 정상 차단 2/5` (❌ "20%"만 제시 금지).
   - **균형 구성** (verifier가 전부 uncertain으로 도망가 점수 따는 문제 검출): 근거 있는 주장 /
     근거 없는 주장 / 근거와 반대되는 주장 / 애매한 주장 / 제안·추론(검증 대상 아님).

### Tier 3 — 실제 사실 검증 (발표 후 로드맵, 문서로만 명시)
9. URL 원문 Extract (§8.4) — Tavily Extract, 비용·지연 큼.
10. **주장별 근거 문장 검색 + 판정** — 검증 단위를 다음 체인으로:
    `최종 기획서 주장 → 지지 표시된 URL → 원문 내 관련 문장 → support/contradict/insufficient`.
    (지금처럼 Research 전체 텍스트와 비교하는 것보다 명확)
11. 출처 최신성·**독립성** 판정 (§8.6) — ⚠️ 도메인만 달라도 동일 보도자료 복제일 수 있어
    자동 판정 난이도 높음.
12. 30~50건 GT Benchmark + 인용 정확도/허위 통과율 정식 평가 (§9, §11).

## 5. 지금 하지 않을 것 (명시적 비목표)

- URL 원문 재접속/Extract — 비용·지연·구현 리스크 대비 발표 효과 낮음.
- 출처 독립성 자동 판정 — 정확도 확보가 어려워 잘못된 확신 유발 위험.
- 대규모 GT — MVP 범위 초과 (요약 §13과 동일 판단).
- **URL 개수를 늘리는 방향의 튜닝** — 요약 §16.10: 핵심 과제는 개수가 아니라
  주장-원문 일치도·허위 통과율. 개수 최적화는 헛수고.
- **URL 수를 신뢰도·정확도 점수로 사용하는 것** — 추적 가능성 지표일 뿐.

## 6. 성공 기준 (이 전략이 맞았는지 판정)

Tier 1의 최우선 기준은 **잘못된 확신을 주지 않는 것**이다.

- [ ] 각 검증 결과에 **검증 범위(verification_scope)** 가 표시됨.
- [ ] `supported`가 "원문 사실 확인"으로 오해되지 않게 표현됨.
- [ ] 각 출처에 **출처 유형**이 표시됨 (❌ "고신뢰" 표현 없음).
- [ ] **UI·내보내기 문서·JSON에 동일한 한계 문구**가 표시됨.
- [ ] URL 수를 신뢰도/정확도 점수로 사용하지 않음.
- [ ] 발표에서 "우리가 검증하는 것 / 못 하는 것"을 1장으로 설명 가능.
- [ ] Q1~Q6에 코드 근거로 답변 가능 (verifier docstring이 이미 정직함을 강조).
- [ ] (여유 시) 허위 통과율 수치 1개 확보 — **n/N 형태**로.

정량 보조 지표: 출처 유형 분류 누락률 · 검증 범위 표시 누락률 ·
"원문 검증 완료"로 오해될 문구의 존재 여부.
