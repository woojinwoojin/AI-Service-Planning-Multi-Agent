# 서비스 품질 평가 루브릭 & 프로세스 v1 (Phase 1 · 개편 前 기준선)

> 로드맵 v2 Phase 1의 산출물. 구조를 크게 바꾸기 *전에* "정말 좋아졌는가"를 측정할
> 고정 평가 기반이다. 개편(Phase 2~) 후에는 **동일 세트·동일 루브릭**으로 전후를 비교한다.

## 왜 별도 루브릭인가

- 발표 비교실험용 5개 기준(`app/services/compare.py` `CRITERIA`)은 **그대로 둔다** — 옛 벤치와의
  직접 비교가 깨지지 않도록.
- 서비스 품질 게이트(Phase 4)를 향한 축은 **8개 기준**으로 새로 정의하고 버전으로 고정한다.
- 루브릭·프롬프트가 바뀌면 `RUBRIC_VERSION`/`PROMPT_VERSION`을 올려 옛 리포트와 섞이지 않게 한다
  (로드맵 상시 트랙 C).

## 8개 기준 (각 0~20점, raw 160 · 100환산 병기)

| key | 라벨 | 채점 기준 |
|---|---|---|
| `problem_definition` | 문제 정의 | 문제가 구체적·실재하며 왜 지금 중요한가 |
| `customer_specificity` | 고객 구체성 | 목표 고객이 세분화되고 상황·니즈가 드러나는가 |
| `market_analysis` | 시장 분석 | 시장 규모·성장성·구조·트렌드가 근거와 함께, PESTEL이 채워졌는가 |
| `competitive_differentiation` | 경쟁 차별성 | 경쟁자를 실제로 파악하고 차별점이 설득력 있는가 |
| `revenue_model` | 수익 모델 | 수익 구조가 명확하고 고객·가치와 일관·현실적인가 |
| `feasibility` | 실행 가능성 | 추진 계획·핵심 기능·위험 대응이 구체적·실행 가능한가 |
| `logical_consistency` | 논리 일관성 | 섹션 간 주장이 모순 없이 문제→해결→시장→수익으로 이어지는가 |
| `evidence_usage` | 근거 활용 | 핵심 주장에 실제 출처·데이터가 붙고 수치가 날조 없이 근거 기반인가 |

정의 위치: `app/services/eval_set.py` `RUBRIC` · 프롬프트: `app/prompts/templates.py` `EVAL_JUDGE`.
(테스트 `test_rubric_keys_present_in_prompt`가 둘의 키 일치를 강제한다.)

## 평가 구성요소

1. **루브릭 채점(LLM 심판)** — 8기준을 심판 **N회(기본 3)** 반복해 기준별 **평균 + 표준편차**를 낸다.
   - `total_stdev`(총점 표준편차) = **심판 변동성 지표**. 클수록 평가가 불안정 → *90점 절대 신뢰 금지*.
2. **결정론적 구조 검사** — `parallel_bench.structural_quality` 재사용(14섹션·순서·빈 섹션·PESTEL 표·출처 수·검증 분포). LLM 없이 측정.
3. **비용·지연** — wall time·토큰·추정 비용·fallback 수.
4. **실험 서명** — git commit·모델·주제 해시·심판 횟수·**rubric/prompt 버전**·temperature 메모.

## 고정 평가 세트

- `app/services/eval_set.py` `TOPICS` — **12개 주제**, 순서 고정(= '시드'). 각 주제에 안정적 `id`.
- 완전 결정론은 아니다(LLM temperature=0.3). 그래서 변동성을 `total_stdev`로 **측정**한다.

## 사람 기준선 보정 (90점 절대 신뢰 금지)

사람이 쓴 좋은 기획서 3~5건을 **같은 루브릭**으로 채점해, 모델 점수가 과대/과소한지 확인하고
임계값을 보정한다.

- 입력 형식: JSON `[{"id": "...", "plan": "<기획서 전문>"}, ...]`
- 실행: `python run_eval.py --human data/human_plans.json`
- 리포트에 **사람 총점 vs 모델 총점 · 차이**가 표로 남는다.
- 해석: 모델이 사람 우수작보다 높게 나오면 심판이 후한 것 → 임계값 상향 또는 프롬프트 강화.

> `data/`는 `.gitignore` 대상이라 사람 기획서 원문은 커밋되지 않는다(개인/저작물 보호).

## 실행

```bash
python run_eval.py                    # 전체 12주제, 심판 3회
python run_eval.py --topics 5 --samples 2
python run_eval.py --fresh            # 이어하기 캐시 무시
python run_eval.py --human data/human_plans.json
```

산출물: `docs/eval_report.md`(사람 판독용) · `outputs/eval.json`(원자료).
중단돼도 `outputs/eval_partial.json`에 주제별로 저장되어 다시 실행하면 이어서 진행한다
(실험 서명이 다르면 자동 폐기).
