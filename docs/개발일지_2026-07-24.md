# 개발일지 — 2026-07-24

## 개요

로드맵 v2의 Phase 2 선택적 재작성부터 Phase 5 State 버전 관리까지 핵심 과제를 연속적으로 구현하였다. 문서 재생성 병목 최적화, 정보 신뢰도 Tier 2, 서비스 품질 게이트와 API·State 호환성 개선을 진행하고, 주요 성능 및 신뢰도 개선 사항을 실측하였다.

총 12개의 PR(#62~#73)을 진행했으며, 작성 시점 기준 #67을 제외한 PR의 병합을 완료하였다. 작업은 작은 단위의 PR 작성, 커밋과 푸시, 사용자 확인 후 병합 순서로 진행하였다. 회귀 테스트는 더미 및 Mock 환경에서 수행하여 불필요한 API 비용을 방지하고, 성능 개선은 벤치마크를 통해 전후 차이를 측정하였다.

> 남은 과제(참고): Phase 2의 Artifact Contract·제한적 동적 Agent 실행, Phase 5의 오류 응답 형식 통일·Streaming 이벤트 계약·OpenAPI 정리, 그리고 사람 기획서 기반 임계값 보정(calibrated_threshold)은 이후 과제로 남겨 두었다.

## 1. Phase 2 — 섹션 단위 수정 (PR-7, #62)

전체 문서를 다시 작성하는 대신 Reviewer가 구조화된 이슈로 지정한 문제 섹션만 Draft Writer의 `section_revise`가 재작성하도록 개선하였다. 이때 기존 Agent 분석 결과와 관련 이슈를 문맥으로 활용하며, 수정 대상이 아닌 섹션은 원문을 그대로 유지한다.

- 14개 섹션을 안정적으로 분리하고 다시 조립하는 `sections.py` 추가(미수정 섹션은 원문 byte 그대로 보존)
- Reviewer 결과에 대상 섹션·심각도·문제 유형·수정 지시를 담은 구조화 이슈 추가
- `_route_revision`을 `section_revise`, 전체 `revise`, `finalize`의 3개 경로로 구성
- 섹션 파싱이나 재조립이 실패하면 기존 전체 재작성으로 전환하는 fallback 유지

> 향후 Issue Router가 문제 섹션의 담당 분석 Agent를 직접 재호출하는 구조로 확장할 수 있다.

## 2. Phase 3 — 정보 신뢰도 Tier 2 (#63, #64)

### #63 주장 유형 및 근거 상태 분리
기획서의 주장을 `fact`, `inference`, `proposal`로 분류하고 사실 주장만 근거 검증 대상으로 처리하였다.

- `unsupported`: 현재 수집된 근거에서 확인되지 않음
- `contradicted`: 반대되는 근거가 확인됨 (기존 '미확인'과 분리)
- `not_applicable`: 추론 또는 제안으로 사실 검증 대상이 아님
- `fact_support_rate`, `evidence_link_rate` 지표 추가
- 기존 Verifier 호출의 출력 스키마를 확장하여 추가 LLM 호출 없이 구현

### #64 Ground Truth 스모크 평가
10건의 균형 Ground Truth 세트와 `judge_claim` 함수를 추가해 실제 Verifier의 주장 분류 및 상태 판정 품질을 측정하였다.

## 3. 문서 재생성 최적화 (#65~#68)

- **#65**: GT·병렬 벤치·서비스 품질 평가를 실측하였다. 실측 과정에서 `section_revise` 시간이 단계별 집계에서 누락되는 문제를 발견하고, 이를 `revise_or_finalize` 단계와 critical path에 포함해 timing coverage를 복원하였다.
- **#66**: 구조가 정상이고 표현 관련 이슈가 없는 경우 전체 문서를 다시 편집하지 않도록 조건부 Polish를 구현하였다.
- **#67**: PR-8 적용 후 성능 측정 결과와 실험 한계를 정리한 리포트를 작성하였다. *(작성 시점 기준 병합 대기)*
- **#68**: Polish 생략본과 실행본을 블라인드로 비교하여 읽기 품질 열화 여부를 확인하였다.

## 4. Phase 4 — 품질 게이트 (#69~#71)

- **#69**: 최종 점수, critical·major 이슈, 문서 구조와 사실 근거 충족률을 결합한 `release_ready` 게이트를 구현하였다. 미통과 사유와 미해결 이슈를 State·API 응답·UI에 표시하였다. 임계값은 사람 보정 전 잠정값으로 명시하였다.
- **#70**: 재작성본의 점수가 초안보다 낮으면 초안을 최종본으로 다시 채택하는 최고 버전 유지 기능을 추가하였다(`_select_best`).
- **#71**: 작성 모델과 별도의 Reviewer 모델을 지정할 수 있도록 지원해 자기 채점 편향을 완화할 수 있는 구조를 마련하였다. 별도 모델을 지정하지 않으면 기존 작성 모델을 사용한다.

## 5. Phase 5 — API 및 State 계약 (#72, #73)

- **#72**: `state_version`과 `migrate.upgrade_state`를 도입하였다. SQLite에 저장된 과거 프로젝트를 조회할 때 신규 필드의 기본값을 보완하고 품질 게이트를 소급 계산하도록 구성하였다. FastAPI 버전을 `0.2.0`으로 변경하였다.
- **#73**: `ARCHITECTURE.md`를 현재 병렬 워크플로, Evidence Registry, 신뢰도 Tier 2, 섹션 단위 수정, 조건부 Polish, 품질 게이트 및 State 버전 관리 구조에 맞게 전면 개정하였다.
